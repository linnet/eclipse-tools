Quickmarks Plugin (Both Styles)

This extension enables you to set, move, delete and jump to text markers by one key stroke only, without having to fill in dialog fields.

As delivered this plugin behaves similar to "B" company style, where the keys 0-9 together with some modifier keys are used to set/move/delete or jump to a mark with that number, switching/opening editors, if neccessary.

The key binding for this style is well known: Shift+Ctrl+# sets/moves/deletes a mark with the number # and Ctrl+# jumps to it (where '#' stands for one of the keys/integers 0-9).

As especially Ctrl+1 is by default bound to the most important JavaEditor "Quick Fix" command, we decided to rebind the "Go to Quickmark" actions to Alt+# (purists might want to look at How to have default "B" company key bindings).

As you might guess from the plugins name, you can change it to behave similar to "M" company style. With this style there are only 3 key bindings for the actions used, you can have multiple marks too, even more than 10. But they are not targetable directly, jumping to them happens by toggling through all existing ones. Jumping around is restricted to the current document also, so there is no editor switching/opening (though mine is lon... errmhh, this plugin can do different and better) and a "move" of a mark is missing too.

The key bindings for that style are also well known: Ctrl+F2 sets/deletes a Quickmark, F2 jumps to the next Quickmark and Shift+F2 jumps to the previous Quickmark.

2 of these 3 key bindings are used by Eclipse 2.1 already, so we suggest (and defaulted to) another binding: Shift+Ctrl+F12 to set a Quickmark, Ctrl+F12 to jump to the next and Shift+F12 to jump to the previous Quickmark. In our installations (with several other extensions installed) these bindings are unused (F12 is used in Standard and Emacs configuration).

If you prefer this style be sure to read switching styles to get it whitout having to re/unbind 21 keys manually.

But note, that this plugin easily combines both styles, to give you the best of both worlds. Read the docs, especially have a look at the plugins preference page to learn, what this plugin can do for you.

How to get it

You probably have done this already, but if not, you can download the plugin at:

  http://eclipse-tools.sourceforge.net/quickmarks/

as a ZIP file.

How to install it

  • Stop Eclipse, if it is running.
  • Remove any previous installation of net.sourceforge.eclipsetools.quickmarks (and, just in case, any de.re.quickmarks* and also any dk.kampstruplinnet.quickmarks*).
  • Copy the ZIP parallel to your Eclipse folder (e.g. if your Eclipse is installed at C:\eclipse then copy the zip to C:\net.sourceforge.eclipsetools.quickmarks_1.x.x.zip) and unzip it. It will install to the correct locations (<ECLIPSE>/plugins/net.sourceforge.eclipsetools.quickmarks_1.x.x) and (<ECLIPSE>/features/net.sourceforge.eclipsetools.quickmarks_1.x.x).
  • Remove the ZIP.
  • Restart Eclipse.
  • Customize all of your perspectives that should have the Quickmarks enabled and check 'Other >> Quickmarks Both Styles' (the Quickmarks are available automatically in all perspectives not yet customized).

Known Bugs and Problems

none with this plugin yet, but:

  • there are several Editors where marks ain't settable at all. Namely the PDE multi page editor for plugin.xml, even in it's "Source" sub page does not allow marks to be set, though it appears to look like a full blown editor and is read/write. Same with the build.properties editor. Another evidence is the read only editor you open by jumping to the definition of some class/instance/method ending up in a class file. And another common case is an editor opened from within the "Plug-ins" view, usually ending up in the source of something packed inside some JAR, not editable and not markable too. It's a pitty, marks of any flavour would make life easier there too, but Eclipse doesn't allow them there. So the Quickmarks plugin falls back to just message the fact to you in the statusline.
  • deleting/cutting text blocks with a (Quick)mark in it behaves somewhat strange: if the mark is at the very beginning of the block, the mark remains at the current line, but now marks irrelevant content. If the mark is somewhere in the middle of the block the mark disappears completely, which is OK when deleting, but I expected the mark to reappear on a paste or undo operation. I don't think, this is my fault, it happens with standard bookmarks too.

Acknowledgments

This plugin was based heavily on the work of Jesper Kamstrup Linnet, who released a simpler plugin for Eclipse 3.x. See also http://eclipse-tools.sourceforge.net/quickmarks/.

I used much of Jespers ideas, style and code. My addition, driven by the fact, that duplicate Quickmarks can occur when closing/reopening Eclipse projects, mainly constitutes in the addition of the Quickmarks preference page and the smart handling of duplicate Quickmarks. The option to have different selections remembered in the marks comes from my old implementation of a "M" company style Quickmarks plugin, never really published. The idea to have this plugin resemble best of both worlds came while implementing and testing this variant.

So this is an extension to Jespers work, intended for 2.1.x, but running under Eclipse 3.0 too!.
